# 任务 2：场景目标检测与视频多目标跟踪实验报告

## 一、任务概述

本次任务围绕场景目标检测与视频多目标跟踪展开，主要目标是在 VisDrone 无人机航拍检测数据集上微调训练一个现代单阶段目标检测模型，并将训练后的模型应用到自采视频中，实现逐帧检测、多目标跟踪、遮挡场景分析以及基于 Tracking ID 的越线计数。

在实验中，我使用 YOLOv8n 作为基础检测模型，并在 VisDrone 数据集上进行了 50 个 epoch 的微调训练，得到实验模型 `visdrone_yolov8n_e50`。随后，我使用训练得到的 `best.pt` 对一段校园道路场景视频进行推理。视频中包含行人、电动车、道路、树木、建筑物和灌木等复杂背景，具有一定的真实场景干扰。模型在视频中能够输出检测框、类别以及跟踪 ID，但也暴露出误检、类别混淆、检测框不稳定和越线计数误差等问题。本文将从数据集、模型训练、视频跟踪、遮挡分析和越线计数几个方面对实验过程与结果进行说明。

## 二、数据集与模型选择

本实验使用 VisDrone 检测数据集作为训练数据。该数据集主要来源于无人机航拍场景，包含大量密集小目标，类别包括 pedestrian、people、bicycle、car、van、truck、tricycle、awning-tricycle、bus 和 motor 等。由于本任务要求在场景目标检测中完成多类别检测与视频跟踪，VisDrone 数据集较适合用于训练面向道路、行人和车辆场景的检测模型。

在模型方面，我选择 YOLOv8n 作为基础模型。YOLOv8n 是 YOLOv8 系列中较轻量的版本，参数量较小，推理速度较快，适合在个人电脑 GPU 上完成训练与视频推理。本实验使用的显卡为 NVIDIA GeForce RTX 4060 Ti，显存约 8GB。考虑到 VisDrone 数据集中目标密集、图像数量较多，如果直接使用较大的模型，训练开销和显存压力都会增加。因此，本实验优先选择 YOLOv8n，在保证能够完成任务流程的前提下获得较高的训练和推理效率。

## 三、实验环境与仓库管理

本实验在已有的 Anaconda 虚拟环境中完成，主要使用 PyTorch、Ultralytics YOLO、OpenCV、NumPy、Matplotlib 等工具。由于之前在训练测试阶段遇到过 DataLoader 内存不足的问题，后续正式训练时采用了较保守的数据加载设置，将 `workers` 设置为 0，并适当降低 batch size，以保证训练过程稳定运行。

项目仓库按照工程化方式进行管理，主要目录包括 `configs`、`scripts`、`datasets`、`runs`、`outputs` 等。其中，`configs` 用于保存训练配置文件，`scripts` 用于保存训练、验证、视频跟踪、越线计数和帧提取脚本，`datasets` 用于存放 VisDrone 数据集，`runs` 用于保存 YOLO 训练与验证结果，`outputs` 用于保存最终输出的视频和遮挡分析帧图像。通过这种结构，训练过程、推理过程和结果整理都能够较清晰地复现。

## 四、模型训练过程

在正式训练前，我先进行了 1 个 epoch 的测试训练，用于检查数据集路径、标注格式、GPU 调用和 YOLO 训练流程是否正常。第一次测试时，由于默认使用了较多 DataLoader workers，在 Windows 环境下进行 Mosaic 数据增强时出现了内存不足错误。错误发生在 DataLoader worker 中，说明问题主要来自多进程数据加载与增强过程，而不是模型本身或 GPU 显存不足。

随后，我将训练参数调整为较稳定的配置，使用 `batch=4`、`workers=0`、`device=0` 重新进行测试训练。测试训练能够完整跑完 1 个 epoch，并完成验证，说明数据集、训练环境和 GPU 设置均已正常工作。在此基础上，我使用 YOLOv8n 在 VisDrone 数据集上进行了 50 个 epoch 的正式训练，训练得到的模型命名为 `visdrone_yolov8n_e50`，最终使用 `best.pt` 作为后续视频检测与跟踪模型。

本实验的主要训练配置如下：

```yaml
model: yolov8n.pt
data: VisDrone.yaml
epochs: 50
imgsz: 640
batch: 4
workers: 0
device: 0
project: runs/detect
name: visdrone_yolov8n_e50
```

训练完成后，我使用 `best.pt` 在 VisDrone 验证集上进行验证，得到整体性能如下：

| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| all | 548 | 38759 | 0.412 | 0.318 | 0.287 | 0.162 |
| pedestrian | 520 | 8844 | 0.398 | 0.342 | 0.297 | 0.119 |
| people | 482 | 5125 | 0.476 | 0.233 | 0.222 | 0.0731 |
| bicycle | 364 | 1287 | 0.204 | 0.0793 | 0.0569 | 0.0195 |
| car | 515 | 14064 | 0.576 | 0.757 | 0.719 | 0.468 |
| van | 421 | 1975 | 0.444 | 0.334 | 0.321 | 0.218 |
| truck | 266 | 750 | 0.440 | 0.292 | 0.276 | 0.180 |
| tricycle | 337 | 1045 | 0.415 | 0.225 | 0.194 | 0.100 |

从结果可以看出，模型对 car 类别的检测效果明显好于其他类别，mAP50 达到 0.719，Recall 达到 0.757。这说明在 VisDrone 验证集中，车辆类目标具有较高的可检测性。相比之下，bicycle 和 awning-tricycle 的效果较差，其中 bicycle 的 mAP50 仅为 0.0569，Recall 仅为 0.0793，说明模型在区分小型非机动车目标时存在明显困难。这也为后续自采视频中出现“自行车或电动车被识别为 truck”等现象提供了依据。

## 五、视频多目标跟踪实现

完成检测模型训练后，我准备了一段校园道路场景视频作为测试视频。视频画面中包含道路、行人、电动车、树木、建筑物、灌木和阴影等元素。该场景与 VisDrone 数据集相比存在一定差异：VisDrone 主要是无人机航拍视角，而本实验视频更接近地面或路边视角；同时，视频中有大量树叶、树影和背景纹理，这些因素会对目标检测产生干扰。

在视频推理阶段，我使用训练得到的 `best.pt` 调用 YOLOv8 的 tracking 功能进行逐帧推理。跟踪算法为 YOLOv8 内置的多目标跟踪方法，并通过 `persist=True` 保持视频帧之间的轨迹连续。每一帧中，模型输出目标检测框、类别名称、置信度以及 Tracking ID。输出视频中可以看到，部分目标能够被赋予稳定 ID，例如某些行人在连续帧中保持相同的 ID，从而满足任务中“为同一目标分配稳定跟踪 ID”的基本要求。

在实际视频结果中，模型也出现了明显的不稳定现象。例如，画面中的树叶间隙、树影或背景纹理有时会被误识别为车辆；一些自行车或电动车目标会被错误识别为 truck；部分检测框会在连续帧之间发生抖动、消失和重新出现。这说明模型虽然能够在 VisDrone 验证集上获得一定检测能力，但在自采校园视频中泛化能力仍有限。

造成这种现象的原因主要有两方面。第一，训练数据和测试视频存在域差异。VisDrone 是无人机航拍数据，目标通常较小且视角偏俯视；而本实验视频中目标更接近地面视角，背景结构和目标外观都有所不同。第二，校园道路视频中存在树叶、阴影、灌木、墙面等复杂纹理，容易形成类似车辆边缘或局部轮廓的视觉模式，从而导致误检。由于跟踪算法依赖检测器输出，当检测框本身不稳定时，Tracking ID 和轨迹自然也会受到影响。

## 六、遮挡与 ID 跳变分析

根据任务要求，我在测试视频中选取了一个发生遮挡或密集交汇的片段，并截取连续 3 到 4 帧进行分析。在该片段中，目标在运动过程中受到其他前景目标或背景结构遮挡，模型输出的检测框出现了短暂丢失。也就是说，在遮挡发生时，检测器没有稳定地在每一帧中识别出目标，导致画面中的目标框发生中断。

不过，从 Tracking ID 的角度观察，该片段中并没有发生明显的 ID 跳变。目标在重新被检测到之后，没有被错误地分配给另一个已有目标，也没有出现两个目标互换 ID 的情况。因此，该遮挡片段的主要问题可以概括为“目标短时丢失，但未发生明显 ID switch”。

这一现象符合多目标跟踪中的常见规律。多目标跟踪通常依赖检测结果、运动预测和帧间匹配来维护目标轨迹。当目标被遮挡时，可见区域减少，检测置信度下降，严重时检测器会直接漏检，导致目标框暂时消失。如果遮挡时间较短，跟踪器仍可能根据历史轨迹和运动连续性在目标重新出现时恢复原有 ID，因此不一定会发生 ID 跳变。但如果遮挡时间更长，或者多个相似目标在同一区域交汇，跟踪器就更容易出现 ID switch 或轨迹断裂。

本实验中的遮挡结果说明，当前模型和跟踪算法在短时遮挡下仍能维持一定的 ID 一致性，但检测器对遮挡目标的召回能力不足，无法保证检测框在每一帧都连续存在。对于实际应用而言，这种目标丢失会影响后续的轨迹分析和越线计数准确性。

## 七、越线计数实现与结果分析

在越线计数部分，我在视频画面中设置了一条虚拟线，并利用检测框中心点与 Tracking ID 的连续性进行计数。具体逻辑是：对于每一个具有 Tracking ID 的目标，记录其上一帧中心点位于虚拟线的哪一侧；如果同一 ID 在后续帧中从线的一侧移动到另一侧，并且该 ID 之前没有被计数，则认为该目标完成一次越线，并将总计数加一。为了避免重复计数，程序维护了一个已计数 ID 集合，同一目标只会被统计一次。

理论上，这种方法能够利用 Tracking ID 的连续性避免同一目标被重复计数。然而，在实际测试视频中，画面中实际有 3 个目标穿过虚拟线，但程序最终统计结果为 1，与真实数量存在差距。这个结果说明，越线计数的准确性不仅取决于计数代码本身，也高度依赖检测和跟踪结果的稳定性。

造成计数偏差的主要原因是检测框不稳定。当目标靠近虚拟线时，如果检测框消失、类别突然改变，或者 Tracking ID 轨迹中断，程序就无法获得完整的“线前位置”和“线后位置”，因此不会把该目标计入穿线数量。此外，检测框在连续帧中抖动也会影响中心点位置，使目标中心点的运动轨迹不够平滑。如果某个目标在越线前后没有保持同一个稳定 ID，或者在关键帧中被漏检，越线逻辑就会失效。

因此，本实验的越线计数结果虽然没有完全达到真实数量，但它清楚展示了基于检测与跟踪的计数方法的工作原理和局限性。对于更高准确率的越线计数，可以考虑提高检测模型性能、调整置信度阈值、优化 tracker 参数、增加轨迹缓存时间，或者在计数逻辑中加入多帧平滑和轨迹插值，以减少短时漏检对最终计数的影响。

## 八、实验问题与改进方向

本实验中最主要的问题是模型在自采视频上的泛化能力不足。虽然 YOLOv8n 在 VisDrone 验证集上经过了 50 个 epoch 的训练，但测试视频与训练数据在视角、背景、尺度和光照方面存在差异，导致模型出现树叶间隙误检为车辆、自行车误识别为 truck、检测框抖动和目标漏检等问题。这些问题进一步影响了多目标跟踪和越线计数结果。

如果继续改进，可以从几个方向入手。首先，可以增加与测试视频更接近的自采数据，并进行人工标注或半自动标注，再对模型进行进一步微调，使模型适应校园道路和地面视角场景。其次，可以尝试更大的模型，例如 YOLOv8s 或 YOLOv8m，以增强特征表达能力，但这会带来更高的训练和推理成本。再次，可以调整推理阶段的置信度阈值和 tracker 参数，在误检和漏检之间取得更合适的平衡。最后，在越线计数部分，可以引入轨迹平滑、短时丢失补偿和多帧判定机制，而不是只依赖单帧中心点变化，这样可以减少框抖动和短时漏检造成的计数错误。

## 九、实验总结

本次实验完整实现了从 VisDrone 数据集训练目标检测模型，到自采视频多目标跟踪，再到遮挡分析和越线计数的流程。训练阶段使用 YOLOv8n 在 VisDrone 数据集上微调 50 个 epoch，最终模型在验证集上取得了整体 Precision 0.412、Recall 0.318、mAP50 0.287、mAP50-95 0.162 的结果。在视频阶段，模型能够对校园道路场景中的部分行人和交通参与者进行检测，并输出 Tracking ID，实现了基本的视频多目标跟踪。

实验也暴露出实际场景中的困难。由于训练数据和测试视频存在域差异，模型在校园视频中出现了误检、类别混淆和检测框不稳定等问题。遮挡分析表明，目标在遮挡时会发生短时丢失，但没有出现明显 ID 跳变。越线计数实验中，实际穿线目标为 3 个，而程序统计结果为 1，说明检测与跟踪不稳定会直接影响基于轨迹的计数任务。

总体而言，本实验不仅完成了目标检测、视频跟踪、遮挡分析和越线计数的基本要求，也通过真实视频中的误差现象展示了多目标跟踪系统在复杂开放场景下的局限性。相比单张图像检测，视频多目标跟踪更加依赖检测稳定性、ID 连续性和场景适应能力，这也是后续进一步优化模型和跟踪算法的重要方向。
